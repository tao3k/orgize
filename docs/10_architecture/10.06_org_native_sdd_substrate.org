* 10.06 Org-native SDD Substrate

This record defines the first Org-native Specification Driven Development
boundary for orgize.  It absorbs OpenSpec's useful artifact-graph and delta
requirement ideas without importing OpenSpec's Markdown directory structure.

** Positioning

Org already has the primitives OpenSpec has to emulate with folders:

- headings provide durable artifact boundaries;
- property drawers provide typed metadata;
- `ID` properties provide stable machine identity;
- links provide semantic edges with labels;
- tags and TODO states support fast sparse-tree and agenda queries;
- checkboxes and progress cookies support local execution state.

Therefore an SDD change in orgize should be a heading, not a directory.  A
large SDD may dispatch child SDD headings, and a child SDD may own a compact
task checklist.  File splitting remains an escape hatch for very large or
cross-repository work, not the default model.

** Node Model

An SDD node is any Org section with either a local `sdd` tag or a local
`SDD_KIND` property.

The stable node properties are:

| Property | Meaning |
|----------+---------|
| =ID= | Stable machine identity.  This should be UUID-shaped or ULID-shaped and must be unique in the visible SDD set. |
| =SDD_KIND= | One of =program=, =capability=, =change=, or =task-group=. |
| =SDD_PARENT= | Semantic parent edge.  Prefer an Org =id:= link such as =[[id:<id>][Capability label]]= so the target is precise and the relationship remains readable. |
| =SDD_CAPABILITY= | Search and reporting dimension for the capability or bounded context. |
| =SDD_SLUG= | Human-readable short key for display and filename candidates. |
| =SDD_STATUS= | Workflow state such as =active=, =blocked=, =done=, or =archived=. |

The =ID= value is not the semantic name.  It is the stable identity.  The
=SDD_PARENT= field is semantic: it is a typed parent edge whose target can be
resolved by orgize and whose label can explain the relationship to an Agent.

** Hierarchy

The expected hierarchy is:

#+begin_src text
program SDD
  -> capability SDD
    -> change SDD
      -> task group / checklist
#+end_src

The hierarchy is logical, not path-driven.  A child SDD can live in the same
file as its parent, or in another Org file later when the host supplies a
multi-file index.  The first orgize slice is document-local and read-only.

** Requirement Shape

SDD requirements use native Org headings:

#+begin_src org
** Requirement: Child SDD dispatch
The system SHALL allow a program or capability SDD to dispatch bounded child
SDD changes.

*** Scenario: Agent resumes child change
- WHEN an Agent queries active SDD work
- THEN orgize SHALL expose the child node, parent edge, progress state, and
  evidence boundary.
#+end_src

Requirements and scenarios stay behavior-facing.  Implementation details belong
in a Design or Tasks section under the same SDD heading.

** First Slice

The first implementation slice is intentionally read-only:

1. project SDD nodes from native Org headings and properties;
2. lint missing or malformed =ID=, =SDD_KIND=, and =SDD_PARENT= fields;
3. lint duplicate visible SDD IDs;
4. lint SDD requirements that lack direct scenario headings;
5. render a compact SDD status tree for Agents.

This slice does not archive changes, write files, maintain a database, or
replace a host planning workflow by itself.  Hosts such as Wendao can later use
the projection to replace blueprint/ExecPlan governance with SDD-backed Org
planning.

** Boundary

orgize owns:

- parsing ordinary Org SDD headings;
- projecting SDD node records and parent edges;
- compact status and lint renderers;
- non-mutating archive or apply plans in later slices.

Hosts own:

- file discovery across repositories;
- git write locks and mutation;
- long-lived storage;
- policy decisions about when an SDD can be applied or archived.

This keeps SDD as a semantic projection over Org instead of turning orgize into
a project-management runtime.
