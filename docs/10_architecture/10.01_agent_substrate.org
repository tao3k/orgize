* 10.01 Agent Substrate

This record captures the parser-v2 architectural direction after the agenda,
lint, and compact repair-report work. The goal is to keep orgize parser-native
while making its semantic projections directly useful to LLM agents.

** Positioning

orgize should not stop at being a lossless Org parser. The parser remains the
substrate, but the future-facing boundary is an Org-native semantic compiler
for agents.

The core consumers are no longer only exporters or human-facing renderers. They
also include agents that must decide what to repair, what to schedule, what to
ignore, and what historical fact is still valid for the current task.

** Documentation Policy

Durable documentation follows the path-first Johnny.Decimal topology in
[[file:../topology.toml][docs/topology.toml]]. The coordinate map is
[[file:../00_index/00.01_jdex.org][00.01 JDex]], with a compatibility entry at
[[file:../index.org][docs/index.org]].

Internal governance, status, release-readiness, planning, and architecture
records should use Org as the canonical source format.

Markdown remains acceptable as a compatibility shell when external tooling
requires it. The current examples are:

- `README.md`, because Cargo, rustdoc, crates.io, and GitHub use it as the
  public package entry point.
- `wasm/README.md`, because the npm package publishes it through
  `wasm/package.json`.

Long-lived project decisions should live in `.org` documents. Markdown
compatibility files should point at those sources instead of carrying separate
governance state.

** Agent-Facing Output

The `orgize lint` compact renderer is the current working model:

- Keep JSON as an explicit machine mode.
- Make the default human/agent text compact.
- Include source location, source line, repair hint, and contract.
- Avoid forcing agents to consume large JSON payloads for ordinary repair
  loops.

Planning and memory projections should follow the same shape. The parser should
produce stable semantic facts and compact decision cards; higher-level runtimes
can decide how to schedule, recall, or execute.

These projections must not invent Org source syntax. They should be derived
from syntax and behavior already recognized by Org mode: headlines, TODO
keywords, priorities, tags, planning lines, active/inactive timestamps,
repeaters, property drawers, ordinary drawers, LOGBOOK records, and standard
agenda semantics.

Calibration should start from the local research-only Org mode checkout under
`.data/org-mode`, especially `doc/org-manual.org`, `lisp/org.el`,
`lisp/org-element.el`, and `lisp/org-agenda.el`. That checkout remains outside
Cargo, CI, tests, and release artifacts.

** Planning Projection

The existing `Document::agenda_entries(&AgendaQuery)` API is the first planning
projection. It already turns Org planning syntax into scheduled, deadline,
closed, active timestamp, warning, overdue, repeater, category, and tag-filtered
rows without mutating `ParsedAst`.

`Document::agent_planning_snapshot(&AgentPlanningQuery)` derives compact agent
cards from those rows and section metadata. This is a projection over official
Org planning semantics, not a new planning syntax. That snapshot should answer
decision-oriented questions:

- What is due now or overdue?
- What is blocked?
- What is stale?
- What is the next safe action?
- Which source line and Org heading justify this decision?
- Which tasks are done, archived, canceled, or intentionally excluded?

The output should be compact cards first, with JSON as an explicit mode.

The wasm demo exposes this as an Agent Planning tab so the browser surface
stays synchronized with the Rust semantic API. The tab renders compact cards
from ordinary Org planning syntax in the sample document.

** Memory Projection

Agent memory is not only storage. It is governed temporal projection.

Org is a better source format than plain Markdown for this because headings,
properties, timestamps, tags, TODO states, drawers, and logs can express
validity, scope, correction, and supersession.

A memory-oriented projection should stay within standard Org constructs:

- Headings provide addressable memory records.
- TODO keywords and tags provide lifecycle state.
- Active and inactive timestamps provide time evidence.
- Property drawers provide key-value metadata without changing Org syntax.
- LOGBOOK or ordinary drawers provide state-change history and notes.

Project-specific property names can be conventions for a consumer, but orgize
should treat them as ordinary Org properties unless an official Org construct
gives them special semantics. Old facts should not disappear silently. They
should become closed, archived, DONE/CANCELED, or superseded by ordinary Org
records so retrieval can prevent stale preferences and corrected facts from
re-entering current decisions.

The formal memory projection API design lives in
[[file:10.02_agent_memory_projection.org][10.02 Agent Memory Projection]]. The
short version is: raw memory records are the parse-faithful substrate, while
agent memory snapshots are the primary API for current-decision use.

The wasm demo exposes this as an Agent Memory tab so browser consumers can see
the compact `MEMxxx` projection beside planning, agenda, semantic AST, and
rendering outputs. Those `MEMxxx` values are output diagnostics, not Org source
syntax.

** Agent Capture Protocol

Capture is no longer modeled as Emacs `org-capture-templates` for a human
operator.  The Agent-era object is an interaction event that the Agent
interprets into a non-mutating Org capture plan.  The durable artifact remains
ordinary Org: headline, tags, property drawer, inactive timestamp, links, quote
blocks, and body text.

The formal boundary lives in
[[file:10.05_agent_capture_protocol.org][10.05 Agent Capture Protocol]].  The
short version is: orgize renders and explains a native Org entry plan; wasm
exposes a stable `capturePlanJson` DTO; org-zhixing or another runtime owns rich
interaction, confirmation, and patch application; Wendao and downstream agents
own search ranking and memory promotion.

** Boundary

Keep orgize focused on parsing and semantic projection.

orgize should not become a full task manager, scheduler, or long-term memory
database. It should provide the typed Org-native facts and compact agent-facing
renderers that those systems can consume.

The clean boundary is:

- Parser: lossless syntax and source-preserving AST.
- Semantic core: typed planning, timestamps, properties, drawers, tags, links,
  diagnostics, and side tables.
- Agent projections: compact lint, planning, memory, and capture plans.
- Runtimes: external systems decide execution, persistence, and scheduling.
