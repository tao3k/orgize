** Durable Consumer Contract

The durable downstream test is:

#+begin_src sh
direnv exec . cargo test --locked --test integration_test git_locked_harness_report_exposes_actionable_performance_gaps
#+end_src

The test builds a small latency-sensitive Rust crate through Orgize's
git-locked harness dependency, writes modular verification reports, reloads the
runtime manifest and receipt JSON, then opens the selected first artifact from
`RustVerificationReportEntryAdvice`.

The contract intentionally checks the report's optimization surface:

- entry action is `load_selection_advice_sidecar`;
- first artifact is `performance_index_json`;
- `artifact_path("performance_index_json")` points to the persisted
  `performance_index.json`;
- `sidecar_path("selection_advice_json")` points to persisted selection advice;
- the selected performance record is `pending`;
- required evidence includes `benchmark_command`, `baseline`, and
  `regression_threshold`.

Those final three evidence keys are the important part: a report that only says
"tests passed" is not enough. The useful report tells the next Agent exactly
which performance evidence is missing before a latency-sensitive path can be
called optimized.

** Report-Derived Optimization Points

1. Preserve the current structural gate as a clean baseline.

   `rust-project-harness --json` currently reports no blocking findings for the
   checked Orgize parser workspace. The wasm package keeps the same build gate
   in the `tao3k/orgize-wasm` submodule, so parser and browser package policy
   remain enforced at their own repository boundaries; it does not mean
   performance work is complete.

2. Promote dense parser paths into explicit performance evidence.

   The parser-v2 closeout already names dense Criterion surfaces in
   `20.02_parser_v2_performance_closeout.org`. The report consumer contract now
   says what each latency-sensitive report must carry before it is actionable:
   benchmark command, baseline, and regression threshold. The next useful
   improvement is to map dense parser surfaces to those evidence keys instead of
   leaving benchmark results only as prose.

3. Keep selection policy upstream.

   Downstream code should not guess report filenames. It should load
   `selection_advice.json` first, then open the `first_artifact.path` chosen by
   `RustVerificationReportEntryAdvice`. This keeps small reports cheap and lets
   broader workspaces load `analysis_profile_json` before heavy payloads.

4. Turn missing evidence into backlog, not noise.

   A pending `performance_index_json` record means "run or attach the benchmark
   evidence", not "the harness failed". For Orgize this translates into a
   backlog item: persist dense-path benchmark receipts with baseline and
   threshold metadata whenever parser-v2 hot paths change.

5. Keep evidence-producing tests below harness structure gates.

   The 2026-05-15 dense-agenda benchmark attempt first failed at build time with
   `RUST-AGENT-PROJECT-005` because `tests/integration/lint_fmt.rs` had grown back into
   a large mixed formatter/lint leaf. The fix was to keep table formatting in
   `tests/integration/fmt_table.rs`, move formatter CLI checks into
   `tests/integration/fmt_cli.rs`, and leave `tests/integration/lint_fmt.rs` as
   the lint-focused owner. That structural split reopened the benchmark path
   without weakening the harness gate.

6. Treat new contract fixtures as their own test owner.

   The `PROPERTY_SCHEMA` registry contract initially added more snapshot tests
   to `tests/integration/lint_fmt.rs`, and the harness repeated
   `RUST-AGENT-PROJECT-005`: the file had grown to 486 effective lines across 22 test
   functions. The repair was to split registry loading, invalid-registry
   diagnostics, JSON Schema shape snapshots, and schema-contract lint advice
   into `tests/integration/lint_property_schema.rs`. This keeps the new
   Agent-facing contract evidence durable without regressing the structural
   test ownership gate.
